R
Read Permission (r = 4)
Files: View file contents
Directories: List filenames with ls
Example: cat file.txt
W
Write Permission (w = 2)
Files: Edit, delete file
Directories: Create/delete files inside
Example: echo "text" > file.txt
X
Execute Permission (x = 1)
Files: Run executable/script
Directories: Enter with cd
Example: ./script.sh

💡 Key Principles

  • 3-Layer System: Owner → Group → Others (in that priority order)
  • Directory Rule: Must have execute (x) permission to access files inside
  • Permission Check: System checks owner first, then group, then others
  • Common Values: 755 (scripts), 644 (files), 600 (private files)

⚙️ chmod, chown & Essential Commands

🔧 chmod - Change Permissions

Symbolic Mode (Fine-grained control)

chmod u+rwx file.txt # Add all permissions for user
chmod g-w,o=r file.txt # Remove group write, set others read-only
chmod a+x script.sh # Add execute for all (user, group, others)

Octal Mode (Set all at once)

chmod 755 script.sh # rwxr-xr-x (standard script)
chmod 644 document.txt # rw-r--r-- (standard document)
chmod 600 private.key # rw------- (private file)

Recursive Operations

chmod -R 755 /project # Apply to directory and all contents
chmod -R u+rwX /shared # X = execute only if already set or directory
👤 chown & chgrp - Change Ownership
chown user:group file.txt # Change both user and group
chown :staff file.txt # Change group only
chown admin file.txt # Change user only
chown -R admin:developers /proj # Recursive ownership change

chgrp developers file.txt # Alternative: change group only
🔍 Information & Diagnostics
ls -l file.txt # View file permissions
ls -ld directory # View directory permissions (not contents)
ls -la # Include hidden files (.files)
getent group # List all system groups
groups username # Show user's groups
id username # Show user/group IDs and memberships

⭐ Special Permissions: SUID, SGID & Sticky Bit

Understanding Special Permissions

Special permissions modify the standard rwx behavior to provide enhanced security and collaboration features.
They appear as s (SUID/SGID) or t (sticky bit) in place of the execute permission.

S
SUID (Set User ID) - Octal: 4000
🎯 What it does: Run file with owner's privileges (not executor's)
📍 When to use: System utilities that need root access
🔍 Visual: 's' replaces 'x' in user permissions
⚠️ Security: High risk - audit regularly
💡 Example: passwd command needs to edit /etc/shadow
G
SGID (Set Group ID) - Octal: 2000
🎯 What it does: Files run as file's group; directories inherit group
📍 When to use: Team projects, shared directories
🔍 Visual: 's' replaces 'x' in group permissions
✅ Benefit: Automatic group ownership for collaboration
💡 Example: /var/mail directory for email sharing
T
Sticky Bit - Octal: 1000
🎯 What it does: Only file owner can delete/rename files
📍 When to use: Shared temp directories, public folders
🔍 Visual: 't' replaces 'x' in others permissions
🛡️ Security: Prevents deletion by other users
💡 Example: /tmp directory for temporary files
🧠 Deep Dive: When & Why to Use Special Permissions

🔐 SUID (Set User ID) - The Privilege Escalator

🤔 What Problem Does SUID Solve?

Imagine you need to change your password. The passwd command must edit /etc/shadow (owned by root), but you're just a regular user. SUID allows the passwd program to temporarily "become" root when you run it.

📋 Real-World SUID Examples:

  • /usr/bin/passwd - Users can change passwords (needs root to edit /etc/shadow)
  • /usr/bin/sudo - Execute commands as other users
  • /usr/bin/su - Switch user identity
  • /bin/ping - Send ICMP packets (needs raw socket access)

⚠️ When NOT to Use SUID:

  • On shell scripts (security risk - can be exploited)
  • On files owned by regular users (defeats the purpose)
  • On world-writable files (anyone can modify and gain privileges)
  • When simpler solutions exist (proper group permissions)

👥 SGID (Set Group ID) - The Team Collaborator

🤔 What Problem Does SGID Solve?

Your development team shares a project directory. Without SGID, files created by different team members belong to their individual groups, making collaboration messy. SGID ensures all files inherit the project's group ownership.

📋 SGID Use Cases:

  • Project Directories - Team members create files with shared group ownership
  • Mail Directories - /var/mail ensures mail files have correct group
  • Shared Applications - Programs that need group-level access to resources
  • Web Directories - Files created by web server maintain proper group ownership

🎯 Perfect SGID Setup Example:

# Create shared project directory
mkdir /shared/webdev
chgrp developers /shared/webdev
chmod 2775 /shared/webdev # SGID + group write access

# Now ANY team member's files inherit 'developers' group!
# Alice creates: touch /shared/webdev/index.html → group: developers
# Bob creates: touch /shared/webdev/style.css → group: developers

🏷️ Sticky Bit - The Deletion Protector

🤔 What Problem Does Sticky Bit Solve?

You have a shared directory where everyone can create files, but you don't want users deleting each other's files. The sticky bit ensures only the file owner (or root) can delete their own files, even in world-writable directories.

📋 Sticky Bit Use Cases:

  • /tmp Directory - Users can create temp files but can't delete others' files
  • Public Upload Directories - Users can upload but can't delete others' uploads
  • Shared Scratch Space - Everyone can write, but only owners can delete
  • Student Directories - In educational environments for homework submissions

🎯 Real-World Sticky Bit Example:

# Look at /tmp - the classic sticky bit example
ls -ld /tmp
drwxrwxrwt 15 root root 4096 Dec 21 10:30 /tmp

# What this means:
# - Everyone can read, write, execute (rwxrwxrwx)
# - BUT the 't' means only file owners can delete their files
# - User1 can create /tmp/user1_file
# - User2 CANNOT delete /tmp/user1_file (even though /tmp is world-writable!)

⚠️ Security Best Practices for Special Permissions

🚨 Security Risks:

  • SUID Files: Can be exploited for privilege escalation
  • World-writable SUID: Anyone can modify and gain privileges
  • SUID Scripts: Shell scripts with SUID are dangerous
  • Unnecessary SGID: Can lead to unintended file access

✅ Security Measures:

  • Regular Audits: find / -perm -4000 2>/dev/null
  • Minimal Use: Only when absolutely necessary
  • Proper Ownership: Ensure correct user/group ownership
  • Monitor Changes: Alert on new SUID files

🔒 Security Best Practices & Pro Tips

✅ Permission Best Practices

🎯 Common Permission Patterns:
  • 644 - Regular files (rw-r--r--): Owner can read/write, others can read
  • 755 - Executable files & directories (rwxr-xr-x): Standard access pattern
  • 600 - Private files (rw-------): Only owner can access
  • 700 - Private directories (rwx------): Owner-only access
  • 775 - Group collaboration (rwxrwxr-x): Team write access

🚨 Security Threats & Prevention

⚠️ Common Vulnerabilities:
  • 777 Permissions: World-writable files = security risk
  • Unmonitored SUID: Can lead to privilege escalation
  • Weak umask: Default 022 may be too permissive
  • Group writable configs: System files should not be group-writable
🛡️ Prevention Commands:
# Find dangerous permissions
find /home -perm 777 -type f 2>/dev/null
find / -perm -4000 -user root 2>/dev/null
find /etc -perm -002 -type f 2>/dev/null

# Set secure umask
umask 0077 # Very restrictive
umask 0022 # Standard security

# Audit file changes
find /etc -newer /tmp/baseline -type f
🎓 Pro Tips & Troubleshooting

✅ Quick Fixes

# Fix "Permission denied" on script
chmod +x script.sh

# Fix directory access
chmod 755 directory

# Restore default permissions
chmod 644 *.txt
chmod 755 */

🔍 Diagnostic Commands

# Check file details
stat filename

# Test permissions
sudo -u user test -r file

# Check effective permissions
namei -l /path/to/file

⚡ Advanced Techniques

# Conditional execute permission (X vs x)
chmod -R u+rwX /project # X only adds execute if already set or directory

# Backup permissions before changes
getfacl -R /project > permissions_backup
setfacl --restore=permissions_backup

# Find files by permission pattern
find . -perm /222 -type f # Find any writable files
find . ! -perm -444 -type f # Find non-readable files
🚨 Common Permission Problems

❌ Problem: "Permission denied" when running script

Cause: Missing execute permission

Solution: chmod +x script.sh

❌ Problem: Can't access files in directory

Cause: Directory lacks execute permission

Solution: chmod +x directory

❌ Problem: New files have wrong group

Cause: Directory needs SGID bit

Solution: chmod g+s directory

❌ Problem: Users delete each other's files in shared folder

Cause: Missing sticky bit

Solution: chmod +t directory

🎯 LPIC-1 Exam Tips

  • Memory Aid: "Read=4, Write=2, eXecute=1" (r-w-x = 4-2-1)
  • Quick Check: Odd numbers (1,3,5,7) = executable permissions
  • Directory Rule: Always remember 'x' needed to enter directories
  • Special Permissions: 4=SUID, 2=SGID, 1=Sticky (just like rwx!)
  • Safety First: Test with non-critical files before exam
